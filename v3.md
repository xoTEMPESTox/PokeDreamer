# PokéWorld v3 — SOTA System 1 + System 2 Dual-Agent Architecture

> **Status**: Planned (not yet implemented). This document outlines the research roadmap for the next generation of PokéWorld — a state-of-the-art RL agent for Pokémon Red combining fast reactive control with deliberate multi-modal strategic planning.

---

## ?? Goal

Build a complete, open-ended Pokémon Red-playing agent that can handle the full diversity of in-game tasks (exploration, battle, puzzle-solving, menu navigation, story progression) through a dual-architecture of:

- **System 1** — Fast, reactive, sub-symbolic control layer (RL + Behaviour Cloning)
- **System 2** — Slow, deliberate, symbolic planning layer (Multimodal LLM-based Planner)

## ?? System 1: Fast Reactive Controller

### Phase 1A — Behaviour Cloning (BC) Warm-Start

A BC policy trained on high-quality human game-play demonstrations, providing a skill prior that System 1 can leverage before online RL kicks in.

**Inputs:**
- Native resolution frames: `160 × 144 × 3` (from emulator)
- RAM state vector: player coordinates, map ID, party HP summary, badge flags, in-battle flag

**Model:**
- ResNet encoder (shared trunk with v2 RSSM encoder)
- BC head: `MLP(latent) ? discrete action logits` (8 actions)
- Loss: Cross-entropy against recorded human actions

**Dataset Target:**
- 10–50 hours of curated human play-throughs covering all major game events

---

### Phase 1B — Custom RL Agent (PPO + GRPO)

An online RL agent trained inside the v2/v3 RSSM imagined rollouts and verified via real emulator evaluation.

**Key Design Decisions:**
- **Policy**: Multi-task Actor-Critic with a task-conditioned head
- **RL Algorithm**: PPO with GAE, with GRPO (Group Relative Policy Optimization) for group-based advantage normalization across diverse tasks
- **Reward Shaping**: Dense curriculum reward from RAM:
  - ?(Pokédex seen/caught count)
  - ?(Badges acquired)
  - ?(Story event flags triggered)
  - ?(Map ID diversity / unique tiles visited)
  - Battle outcome: win/loss/catch (+/-)
  - XP gain proxy
- **Exploration Bonus**: Novelty bonus from RSSM prior vs posterior surprise (information gain)
- **Wide Task Coverage**: The agent must generalize across all game states:
  - Overworld navigation
  - Battle turn selection
  - Menu navigation (items, Pokémon party)
  - NPC dialogue advancement

---

### Phase 1C — World Model (v3 RSSM)

An improved discrete RSSM with higher fidelity and richer observation inputs:

| Component | v2 | v3 |
|---|---|---|
| Resolution | 160x144 | 160x144 (native) |
| Latent type | Discrete (32x32) | Discrete (64x32) + Structured RAM tokens |
| Context | Pixels only | Pixels + RAM state vector |
| Imagination horizon | 15 steps | 50 steps |
| KL balancing | 80/20 | Adaptive free-bits |
| Model size | ~650MB | Compact (~200MB, efficient CNN) |

## ?? System 2: Multimodal LLM Strategic Planner

### Role
The LLM Planner operates at a **macro timescale** — it does not select individual button presses. Instead, it:
1. Receives a high-level game-state summary every N real-world seconds (or every M in-game steps)
2. Produces a **goal string** and **sub-goal queue** that System 1 is conditioned on
3. Monitors progress and adjusts the macro strategy when System 1 gets stuck

### Inputs to Planner
- Screenshot of current screen (PNG)
- Structured game-state report (JSON from RAM): player name, coordinates, map, badges, party HP/level/species, inventory key items
- History of last 5 high-level actions / outcomes (natural language summary)
- Failure signal if System 1 has not made progress in K steps

### Outputs from Planner
- **Current Goal** (e.g., "Reach Pewter City Gym and defeat Brock")
- **Sub-goal Queue** (e.g., ["Exit Pallet Town north", "Navigate Route 1", "Enter Pewter City", "Enter Gym"])
- **System 1 Conditioning Token** — a learned embedding of the current goal, injected into the policy head

### LLM Backbone Options (in priority order)
1. **Gemini 2.0 Flash** — low latency, grounded vision reasoning, free tier available
2. **GPT-4o mini** — strong game reasoning, cheap API
3. **Llama 3.2-Vision (local)** — fully offline option via Ollama

### Planner Frequency
- Default: every **300 in-game frames** (~5 seconds of real gameplay at 60fps)
- Failure mode (stuck detection): trigger replanning after **500 steps** with no badge/flag progress

## ?? Key Research Questions

1. **Goal Conditioning Efficiency**: Can a learned goal embedding from the LLM planner reliably shift System 1 behaviour without retraining?
2. **World Model Generalization**: Does a RAM-conditioned RSSM generalize to indoor environments (gyms, buildings) not seen during training?
3. **GRPO vs PPO**: Does GRPO's group-based advantage normalization lead to faster convergence in the highly multi-task Pokémon RL setting?
4. **Planner Hallucination**: How often does the LLM planner produce incorrect game-state reasoning, and what mitigation is needed?
